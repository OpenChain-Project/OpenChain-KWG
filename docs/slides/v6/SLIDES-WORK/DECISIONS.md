## 핵심 작업 원칙

### Source of Truth (인용 우선순위)

작업 중 인용·근거의 우선순위:

1. `content/ko/guide/opensource_for_enterprise/` — Part 2 본문 토대
2. `content/ko/guide/iso5230_guide/` — Part 1 ISO 5230 조항별
3. `content/ko/guide/iso18974_guide/` — Part 1·2-3 보안 보증
4. `content/ko/guide/iso42001_guide/` — Part 3 AI 거버넌스 (2026-04 이후 신설)
5. `content/ko/guide/templates/` — Part 2-2·2-3 정책·프로세스 샘플
6. `content/ko/guide/tools/` — Part 2-4 도구
7. `content/ko/guide/EVIDENCE-CHECK.md` — 입증자료 50개 충족 강도·근거
8. `content/ko/guide/CRITIC-REPORT.md` — 가이드 자체 약점·강점
9. `content/ko/guide/STYLE-REPORT.md` — 통일성 규칙

**금지**: 2026-04 이전 기존 슬라이드 본문을 인용 근거로 사용 (격차 분석에서 "변경 없음"으로 분류된 슬라이드만 어휘·구조 차용 허용).

### 통일성 규칙

다음 규칙은 가이드 작성 규칙(`.claude/rules/guide-writing.md`)·`STYLE-REPORT.md`에서 가져와 슬라이드에도 그대로 적용:

- 종결체 통일 (가이드와 동일)
- alert color 매핑 (`<Callout variant="...">`의 variant 종류 일치)
- 약어 표기 (첫 등장 시 풀네임 병기)
- ISO 인용 형식: `ISO/IEC 5230 §3.4.1.2` 형식
- 코드 블록 언어 라벨 명시 (bash·yaml·json·diff 등)
- 내부 링크는 상대 경로
- Mermaid 다이어그램 스타일 규칙

### 작업 흐름 안전 장치

- 한 항목 완료 시 즉시 `TODO.md`의 체크박스 갱신 + `PHASE.md`의 *마지막 완료 작업* 갱신
- Phase 종료 시 `PHASE.md` 갱신 + git commit 권장
- 결정 사항 변경 시 이 파일의 *변경 이력*에 (이전값 → 신값 + 날짜 + 사유) 누적

### 파일 수정·승인 정책

- Phase 3 (슬라이드 재작성)는 **파트 단위 묶음 승인** — 한 파트 diff 전체 제시 → 사용자 승인 → 일괄 적용
- Phase 1 (컴포넌트) / Phase 2 (기획) / Phase 4 (페이싱)는 산출물 단위 승인 — 산출물 작성 후 사용자 확인
- 컴포넌트 변경이 슬라이드 본문 회귀를 일으킬 위험이 있으면 Phase 1 완료 후 컴포넌트는 *동결* — 부득이한 변경은 변경 이력에 기록
