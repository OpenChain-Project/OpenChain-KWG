# 가이드 작성 규칙

가이드 관련 작업 시 항상 적용되는 규칙.
대상 디렉토리: `content/ko/guide/` (archive/ 제외)

---

## 파일 읽기 — 토큰 효율화

- 파일 읽기는 목적에 따라 범위를 최소화한다.
  - 구조 파악 → 프론트매터 + 첫 번째 `##` 제목까지만 읽기
  - 조항 내용 확인 → 해당 섹션(##)만 읽기
  - ISO 레퍼런스 → `content/ko/guide/.claude/reference/` 에서 필요한 조항 번호만 발췌
- 본문 전체를 읽기 전에 먼저 구조를 파악하고, 필요한 섹션만 읽는다.

---

## 파일 수정 — 승인 절차

- 파일 수정 전 반드시 diff 형식으로 변경 초안을 먼저 제시한다.
- 사용자 승인("진행해줘", "ok", "적용해") 없이는 어떤 파일도 수정·생성하지 않는다.
- 한 번에 하나의 섹션(##)만 처리한다.
- 기존 문서의 문체, 용어, ISO 조항 참조 형식을 그대로 따른다.

---

## 교차 참조 일관성 — 수정 시 영향 범위 확인

수정 대상 파일에 따라 아래 연관 파일을 반드시 확인한다.

| 수정 대상 | 확인해야 할 연관 파일 |
|-----------|----------------------|
| `iso5230_guide/` 조항 | `opensource_for_enterprise/` 대응 섹션의 교차 링크 |
| `iso18974_guide/` 조항 | `opensource_for_enterprise/` 대응 섹션 + 준용 여부(ISO 5230 대응 조항) |
| `opensource_for_enterprise/` 섹션 | 대응 ISO 조항 가이드의 "참고" 링크 |
| `templates/1-policy/` | 연관 입증자료 조항 가이드의 "샘플" 섹션 |
| `templates/2-process-template/` | 연관 입증자료 조항 가이드의 "샘플" 섹션 |
| `tools/` 페이지 | `opensource_for_enterprise/4-tool` 의 해당 도구 링크 |

## 가이드 간 의존 관계 (요약)

- **ISO 18974 전용 항목** (ISO 5230 준용 없음): `4.1.2.3`, `4.1.2.5`, `4.1.2.6`, `4.1.4.2`, `4.1.4.3`, `4.1.5.1`, `4.2.2.3`, `4.3.2.1`, `4.3.2.2`
- ISO 18974 나머지 항목은 ISO 5230 대응 조항을 준용하므로, 5230 가이드 수정 시 18974 가이드도 영향권에 포함됨
- Templates는 다수 입증자료의 샘플 기반: 템플릿 수정 시 연관 조항 가이드의 샘플 섹션과 정합성 유지

---

## 작성 품질 기준

- 각 조항 페이지는 다음 순서를 따른다:
  1. 조항 개요 (왜 이 조항이 필요한가)
  2. 해야 할 활동
  3. 요구사항 및 입증자료 표
  4. 입증자료별 준수 방법 및 샘플
  5. 기업 오픈소스 관리 가이드 연결 링크
- 입증자료 번호는 원문 그대로 사용 (예: `3.1.1.1`, `4.1.2.3`)
- ISO 조항 번호 병기 형식: `ISO/IEC 5230 §3.4.1.2`
- 내부 링크는 상대 경로 사용

## 통일성 규칙 (Phase 5 검토 결과 기반, 2026-05-12 추가)

### 종결체

- **본문**: 평서체 (`~한다`, `~된다`) — ISO 가이드 조항 페이지
- **본문**: 경어체 (`~합니다`, `~할 수 있습니다`) — 메인 가이드(opensource_for_enterprise) + templates
- **구축 단계 alert**: 본문이 평서체이더라도 alert는 경어체 (`구축합니다`)
- **정책·프로세스 샘플 코드 블록**: 자리표시자(`[회사명]`·`[Year]` 등) + 경어체 — 실제 제출용 문서 모사
- **한 페이지 내 종결체 혼용 금지** — 정책 샘플(코드 블록) 외 본문은 단일 종결체

### Alert color 의미 규칙

- **`info`**: 구축 단계 안내·연결 안내·참고 자료 (예: "Phase N — 단계에서 구축합니다")
- **`success`**: 권장·팁·ISO/IEC 5230 인용 박스·다른 ISO 표준과의 통합·연계 안내
- **`warning`**: 주의·ISO/IEC 18974 인용 박스 (보안 강조)
- **`pageinfo`**: 긴 절차 안내·중요 박스 (예: 인증 절차 3단계)

### 헤딩 번호 체계

- **H2**: `## N. 제목` — 모든 페이지가 번호 부여 (개요부터 참고까지 연속)
- **H3 (그룹별)**:
  - `opensource_for_enterprise/`: `### (N) 제목` (예: `### (1) 오픈소스 라이선스 컴플라이언스 원칙`)
  - `iso5230_guide/`·`iso18974_guide/`: `### N.N.N.N 제목` (입증자료 번호 단위)
  - `iso42001_guide/`: `### §X.X 제목` (조항 번호 단위)
  - `templates/1-policy/`: `### N.N 제목`
  - `templates/2-process-template/`: `### (N) 제목`
- **5섹션 구조 (조항 페이지)**: `## 1. 조항 개요` → `## 2. 해야 할 활동` → `## 3. 요구사항 및 입증자료` → `## 4. 입증자료별 준수 방법 및 샘플` → `## 5. 참고` — 추가 섹션이 필요하면 §4 하위로 흡수
- **6섹션 변형 허용**: 다음 경우 `## 5. 부가 가이드 제목` + `## 6. 참고` 형태로 6섹션 허용
  - 자동화 도구·CI/CD 활용 등 입증자료와 관련된 부가 가이드 (예: `iso5230_guide/3-content-review/2-license-compliance` §5 "자동화 도구 활용", `iso18974_guide/1-program-foundation/5-standard-practice` §5 "자동화 도구 활용")
  - 인증 방법 선택 등 별도 가이드 필요 항목 (예: `iso5230_guide/6-conformance/1-conformance` §5 "인증 방법 선택 가이드")
  - 변형 적용 시 §6에 "참고" 위치를 일관 유지
- **신설 섹션**(예: 7-ai-compliance·4-operation/_index)도 그룹 표준 헤딩 체계 준수

### 약어 풀이

- **첫 등장 시 1회 풀어쓰기** + 약어 병기, 이후 약어만
  - 예: `SBOM(Software Bill of Materials)` → 이후 `SBOM`
- **필수 풀이 대상**: SBOM·OSPM·OSPO·OSRB·SCA·CVE·CVSS·CVD·NVD·OSV·VEX·EPSS·KEV·CRA·GPAI·LLM·MAU·LMS·ML-BOM·AIMS·OSAID
- **페이지마다 재정의 금지** — 그룹 내 첫 등장 페이지에서 풀이, 다른 페이지는 약어만 사용 (또는 약어 표 참조)

### 이미지·캡션

- **표준 표기**: `{{< imgproc 파일명 Fit "900x600" >}}` shortcode 사용 권장
- **caption 형식**: `<center><i>출처: URL</i></center>` 또는 `<center><i>제목</i></center>`
- **alt text 필수**: `![설명](파일.png)` — 빈 alt(`![]()`) 금지
- **raw `![](./파일.png)`** vs **imgproc shortcode** 한 파일 내 혼용 금지 — imgproc로 통일 권장
- **Fit 크기**: `900x600` 표준 (특별한 사유 시 `600x300`·`1200x900` 허용)

### ISO 조항 인용 형식

- **본문 내 조항 참조**: `ISO/IEC 5230 §3.4.1.2` (콜론·괄호 없이 공백)
- **본문 끝 괄호 인용** (templates 패턴): `(ISO/IEC 18974 §4.3.2.1)`
- **첫 등장 시 버전 명시**: `ISO/IEC 5230:2020 (버전 2.1)` → 이후 `ISO/IEC 5230`
- **18974 버전**: `ISO/IEC 18974:2023` → 이후 `ISO/IEC 18974`
- **alert 박스 영문 원문**: 정확한 표준 번호 + 영문 원문 + `<br>` + 한글 번역(백틱 감싸기)

### 용어 통일 사전

| 영문 | 한국어 표기 (통일) | 비고 |
|------|---------------|------|
| Documented Evidence | 문서화된 증거 | ISO 18974 "evidence pack" 의미 강조 |
| documented procedure | 문서화된 절차 | "절차"·"문서화한 절차" 금지 |
| Supplied Software | 공급 소프트웨어 | "제품"·"제품과 서비스" 금지 |
| Open Source | 오픈소스 | 본문 사용 |
| OSS | (본문에서는 "오픈소스") | 표 안에서만 약어 허용 (공간 제약) |
| open source policy | 오픈소스 정책 | |
| Known Vulnerability | 알려진 취약점 | |
| Newly Discovered Vulnerability | 새로 발견된 취약점 | |
| 회사 자기 호칭 | `[회사명]` 자리표시자 | "회사"·"당사"·"OO 회사" 혼용 금지 |

**띄어쓰기 통일**:
- "보안 취약점" (✅) / "보안취약점" (❌)
- "라이선스 컴플라이언스" (✅) / "라이선스컴플라이언스" (❌)

### 코드 블록·shortcode

- **샘플 코드 블록**: 정책·프로세스 샘플은 ```` ``` ```` 무태그 (복사용)
- **실행 코드 블록**: 언어 태그 명시 — `bash`·`yaml`·`groovy`·`text` 등
- **shortcode**:
  - `{{% alert ... %}}` 페어 마커 짝맞춤 필수
  - `{{< imgproc ... >}}` self-closed 형식
  - `{{< pageinfo >}}` 긴 안내 박스

### 링크 표기

- **내부 링크**: 상대 경로 `../../iso5230_guide/3-content-review/1-sbom/`
- **fragment 링크**: `(#섹션-제목)` 형식 (한글 헤더는 소문자·하이픈 자동 생성)
- **외부 링크**: `[표시 텍스트](https://...)` — placeholder 또는 자리표시자 사용 시 명시적으로 `{회사 URL}` 형태
- **금지**:
  - 빈 링크 `[텍스트]` (URL 없음)
  - 이중 대괄호 `[[텍스트]](URL)`
  - 자리표시자 `(insert_link)` — 실제 URL 또는 일반화된 표현으로 교체

### Mermaid

- **방향**: `flowchart TD` (위→아래) 표준 또는 `flowchart LR` (좌→우, 작은 다이어그램)
- **노드 명명**: 영문 ID + 한글 라벨 (`A[정책]`·`B[프로세스]`)
- **노드 모양 의미**: 둥근 사각형(`([ ])`) = 시작/종료, 사각형(`[ ]`) = 처리, 마름모(`{ }`) = 분기
- **줄바꿈**: 노드 라벨 내 `\n` (Docsy 기본 mermaid 지원)
- **범례 권장**: 다이어그램 직후 1줄 캡션 (예: `*둥근 사각형 = 시작/종료 · 마름모 = 분기*`)
